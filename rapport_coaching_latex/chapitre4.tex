\section{Introduction}
Après avoir solidifié la base architecturale et sécurisé l'accès à la plateforme lors du Sprint 1, ce deuxième sprint est entièrement consacré au cœur de la valeur métier de "CoachingApp". Ce chapitre détaille la réalisation des modules de planification d'entraînement (Workout Builder) et de suivi nutritionnel. Nous aborderons la conception du modèle de données hiérarchique complexe nécessaire à la gestion des programmes, ainsi que les défis techniques inhérents au développement du lecteur de séance interactif (Workout Player) sur l'application mobile Flutter.

Les tâches sélectionnées pour ce sprint sont les suivantes :

\section{Réalisation technique : Module Entraînement}

\subsection{Gestion des Programmes (Côté Coach)}
L'interface de création de programme est le défi ergonomique de l'application. Elle permet au coach de construire visuellement un programme d'entraînement complexe.

\subsubsection{Interface de création (UI/UX)}
Sur l'écran, le coach dispose d'une vue en accordéon (ExpansionTile). Il peut cliquer sur "Ajouter un Jour". À l'intérieur de ce jour, il clique sur "Ajouter un Exercice". Un menu modal (BottomSheet) s'ouvre, interrogeant l'API pour récupérer la liste des exercices du catalogue (avec barre de recherche fonctionnelle). Une fois l'exercice sélectionné, des champs de texte apparaissent pour saisir le nombre de séries (sets), les répétitions cibles (reps), le poids indicatif et, élément crucial, le temps de repos (restTimer) en secondes.

\subsubsection{Sauvegarde transactionnelle Backend}
Le JSON généré par le frontend mobile est profond (nested JSON). 
Pour l'enregistrer proprement dans PostgreSQL, nous utilisons l'option `cascade: true` de TypeORM, qui permet d'insérer le Programme, ses Jours et ses Exercices en une seule ligne de code côté service.

\begin{lstlisting}[language=JavaScript, caption={Sauvegarde en cascade d'un programme (TypeORM)}, label={lst:cascade_save}]
async createFullProgram(coachId, athleteId, programData) {
  // programData contient l'arbre JSON complet envoyé par Flutter
  const newProgram = this.programRepository.create({
    ...programData,
    coach: { id: coachId },
    athlete: { id: athleteId }
  });
  
  // TypeORM sauvegarde automatiquement ProgramDay et ProgramExercise 
  // gràce à la configuration @OneToMany(cascade: true)
  return await this.programRepository.save(newProgram);
}
\end{lstlisting}

\subsection{Workout Player (Côté Athlète)}
Le Workout Player est la fonctionnalité différenciante de CoachingApp. Il remplace le "papier et crayon" par une interface dynamique.

\subsubsection{Logique de Chronométrage Asynchrone}
Plutôt qu'une simple liste de contrôle, le Workout Player affiche un exercice à la fois, avec une grande vidéo ou image GIF de l'exécution.
Lorsque l'athlète termine une série, il coche la case correspondante. Cela déclenche un Timer asynchrone en Dart (Flutter). 

\begin{lstlisting}[language=Java, caption={Logique de Timer en Flutter (Dart)}, label={lst:dart_timer}]
Timer? _restTimer;
int _remainingSeconds = 0;

void startRestTimer(int durationInSeconds) {
  setState(() {
    _remainingSeconds = durationInSeconds;
    _isResting = true;
  });

  _restTimer = Timer.periodic(Duration(seconds: 1), (timer) {
    if (_remainingSeconds > 0) {
      setState(() { _remainingSeconds--; });
    } else {
      timer.cancel();
      playBeepSound(); // Alerte sonore de fin de repos
      setState(() { _isResting = false; });
    }
  });
}
\end{lstlisting}

Ce chronomètre aide l'athlète à respecter l'intensité voulue par le coach. De plus, il peut modifier le "Poids soulevé" (Actual Weight) directement sur la série en cours, créant un \texttt{WorkoutLog} qui sera synchronisé avec le serveur, permettant au coach d'évaluer la surcharge progressive de son client.

\section{Réalisation technique : Module Nutrition}
La nutrition représente 50\% des résultats physiques. Nous avons donc intégré un système de suivi diététique poussé.

\subsection{Architecture des Macros}
Chaque aliment consommé possède 4 caractéristiques : Protéines (g), Glucides (g), Lipides (g) et Calories totales.
Le modèle de données \texttt{DietPlan} impose une cible (ex: 2500 kcal, 150g Protéines). 
Lorsque l'athlète ajoute un \texttt{MealLog} (Journal de repas), l'API Node.js recalcule instantanément le total consommé pour la journée en cours.

\subsection{Tableau de bord visuel (Nutrition Dashboard)}
Pour que ces données soient compréhensibles, nous avons utilisé la bibliothèque \texttt{fl\_chart} sous Flutter. Elle permet de générer des "Pie Charts" (graphiques circulaires) animés.
\begin{itemize}
    \item Un graphique principal montre les calories consommées par rapport aux calories totales allouées.
    \item Une jauge de progression linéaire (LinearProgressIndicator) est affichée pour chaque macronutriment (Protéines, Glucides, Graisses). Si l'athlète dépasse l'objectif, la barre change de couleur (ex: du vert au rouge) pour l'alerter visuellement.
\end{itemize}
